项目定义

PR Agent
在解决什么问题？

真正缺少的，是一个能进入实际工作流程、把分散任务串起来、让输出结果可控的工具。

真实工作流里的问题

❌ 现有工具的局限 ✅ PR Agent 的解法
通用 AI 没有品牌规则约束，每次生成结果不一致，术语随机跑偏
引入术语表与规则约束机制，将品牌写法沉淀为可复用系统能力
生成、检查、修改分散在不同工具，流程衔接低效
四类任务整合在同一工作台，选任务 → 输入 → 运行 → 查看结果
通用聊天框让用户不知道从哪开始、该输入什么
按任务分流设计入口，每类任务有独立交互逻辑，降低使用门槛
AI 不认识你的资料库，输出内容和项目背景脱节
RAG 链路支持资料驱动生成，知识库可补充、索引可重建
🎯
核心判断

PR Agent 的目标是让整个处理过程更稳定、更可用、更可控。输出质量的一致性，比生成速度更重要。

产品设计判断

为什么这样设计？

每一个设计决策背后都有具体的业务判断，不是功能堆砌。

🗂️
任务分流，不用聊天框
将高频任务拆解为 generate / check / normalize / rewrite 四类，每类设计独立交互逻辑。

判断依据：不同任务的输入形态、规则依赖和输出要求差异很大，强行统一到一个聊天框只会提高使用门槛，让用户不知道该输入什么。
📋
规则系统，不靠 Prompt 临时注入
引入术语表与规则约束机制，将品牌写法、表达口径与审稿标准沉淀为可复用系统能力。

判断依据：每次靠 Prompt 临时注入规则，输出结果不稳定。把规则固化到系统层，才能保证跨任务、跨用户的输出一致性。
🔍
RAG 驱动，不靠模型"编"
文档解析 → 切分 → Embedding → FAISS 索引 → 语义检索，支持资料驱动的内容生成。

判断依据：汽车公关内容必须有据可查，纯靠大模型生成的内容容易"幻觉"出错误的技术参数或品牌表达。
🎛️
高级功能后置，主流程前置
将术语表管理、参数调整放到侧边栏，首页只保留核心任务入口。

判断依据：功能越多，用户越容易迷失。主流程要让用户第一次进入就知道该从哪开始，复杂功能按需展开。
📌
产品阶段判断

当前处于 Alpha 阶段，重点已从"功能跑通"转向"输出质量优化与交互体验打磨"。这是一个主动判断，不是能力边界——继续堆功能不是当前最有价值的事。

技术实现

把资料、任务与输出
串成一条可复用的流程

📥 资料进入系统的路径

1

上传资料

历史稿件、产品资料、术语表，支持文件上传与知识库补充，也支持临时文件注入。

2

解析与切分

文档解析后按语义边界切分，不硬切句子，保证每个片段语义完整。

3

Embedding 与索引

文本向量化后存入 FAISS 索引，支持语义检索——搜"品牌调性"也能找到写"品牌声音"的段落。

⚙️ 任务执行的路径

4

选任务 · 输入需求

从 generate / check / normalize / rewrite 中选择当前任务，填入需求与上下文。

5

RAG 检索 · 规则注入

从知识库检索最相关的资料片段，同时加载术语表与规则约束，作为生成的依据。

6

生成 · 查看 · 导出

结果直接展示在工作台中，支持左右对照修改、继续优化、导出不同格式。

案例 · CLA 冬季长测

技术测试数据 → 跨媒介内容表达

任务

围绕全新纯电 CLA 冬季长途出行场景，将技术测试数据转化为视频脚本与品牌推送双重输出。

处理方式

以"冬季灵魂拷问"为框架，用户问题前置、参数后置，降低技术内容阅读门槛。视频负责节奏，推送负责解释，同一主题，双重分工。

这个经历说明了什么：技术信息的转化需要理解不同媒介的信息结构，知道什么先说、什么后说、什么不说。这个判断能力，成为了 PR Agent 中 generate 和 rewrite 模式设计的核心参考。
