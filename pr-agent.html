What it is

汽车公关顾问的 内容工作台

PR Agent 不是一个通用写稿机器人，而是围绕汽车传播任务拆出来的垂直 Agent。

它从 brief 进入，把模糊需求整理成传播方向、标题、导语、新闻稿段落和社媒文案；也能对已有稿件做风险检查、口径统一和 PR 风格改写。

我希望它更像真实顾问会用的工作台：输入有结构，输出像文档，过程能解释，关键资料能回到知识库检索，而不是把所有工作都塞进一个空白聊天框。

Workflow

从传播需求到 可编辑初稿

核心流程从 brief 开始，而不是从 prompt 开始。

01传播 brief输入项目背景、目标和初步需求。
02分类字段选择内容类型、品牌、语气与任务目标。
03知识库检索从本地资料中找可引用的上下文。
04策略骨架先搭传播逻辑，再生成具体文案。
05文档交付按章节输出，便于继续修改。
Product Logic

三个模块，对应三种 真实任务

生成、审校、检索的输入不同，判断标准也不同，所以不混成一个聊天入口。

Module 01

传播内容生成

把传播 brief 转成主题、方向、标题、导语、新闻稿段落和社媒文案，先搭策略骨架，再生成具体内容。

Module 02

稿件审校改写

检查已有稿件里的事实风险、时间表述、品牌口径和 PR 语气，给出可直接替换的修改建议。

Module 03

知识库检索

把本地资料变成可查询的上下文，用在生成和审校前，减少空泛表达和无依据判断。

显式字段内容类型、品牌、语气、目标让用户自己选择，避免所有意图都藏在 prompt 里。
文档结果输出按章节整理，方便复制、审阅和二次修改，不像聊天记录。
低 AI 感界面语言克制，强调工作流和交付物，而不是一直提醒用户“这是 AI”。
Engineering Diagnosis

我修掉的一个 检索缺陷

这个项目不只是把页面做出来，也包括读懂系统为什么不稳定。

测试知识库检索时，我发现约三分之一的资料怎么搜都召不回。最后定位到原因：车型名过滤用了过宽的子串匹配，导致部分资料被错误排除。

≈35%知识库内容被静默漏掉，生成时拿不到应有上下文。
0修复后该类漏召回归零，相关检索能正常返回。
零跑回归检查，没有误伤其它车型资料。

Before / substring match

搜索 CLA 时，S-Class 里的 “cla” 也被命中，于是 S 级资料被误伤丢弃。

Incorrect exclusion

After / word boundary

改为识别完整车型词边界后，CLA 和 S-Class 各归各，不再互相污染。

UX Iteration

从功能列表，改成 工作台

最近一版把 Figma Make 生成的 React 前端接到 FastAPI 后端，同时重新调了输入区和交互状态。

第一版更像“功能集合”：生成、检查、统一、改写、检索、上传都并列摆着。现在更像一个可以工作的页面：左侧是任务模块，中间是 brief 表单，结果区按文档结构呈现。

生成检查统一改写检索上传导出设置

问题不在功能少，而在新用户必须先理解每个入口，才知道自己该从哪一步开始。

Build & Delivery

从想法到 能点开体验

我把它当成一个可以交付的产品原型，而不是只停留在 demo。

FrontendReact + Vite，复刻 Figma Make 的交互视觉，并接入真实后端接口。
BackendFastAPI 提供生成、审校、知识库检索接口。
RetrievalFAISS + sentence-transformers，用本地知识库给内容生成提供上下文。
DeployDocker + Zeabur，演示环境使用脱敏或示例资料，不提交真实客户信息。

这个项目想展示的

我可以从真实汽车 PR 工作流里拆问题，也能把它做成可用产品：定义范围、设计交互、定位工程问题、接好前后端，并把它部署到线上。
